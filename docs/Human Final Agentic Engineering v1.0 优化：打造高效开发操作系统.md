# Human Final Agentic Engineering v1.0 优化：打造高效开发操作系统

# 人类终审式 Agentic Engineering 开发体系 v1.0 优化定稿

本版本**100%对齐Peter Steinberger公开实战方法论**，融合两版内容的核心精华，彻底剔除冗余重复、解决流程冲突，专为个人Builder打造「高产能、低内耗、强可控」的开发操作系统。核心目标：帮你彻底从逐行编码的执行者，切换为**定方向、划边界、控风险、做终审**的决策者，实现「Agent扛下99%执行工作，你只抓最高杠杆的核心决策」。

---

## 0. 核心定位与不可逾越的权责边界

这不是一套「AI替你做决定」的系统，而是一套**人机分工极致明确、风险完全可控、产能拉满**的开发体系。

### 你的角色（永远在驾驶位，不可放权的5件事）

你全程不做低杠杆的编码执行，只聚焦核心决策：

1. 产品意图与目标定义（要解决什么问题、不解决什么问题）

2. 架构边界与技术选型（定规则、划红线、拍板实现方案）

3. 风险分级与模式选择（给任务定风险等级，选执行模式）

4. 关键变更终审放行（只看核心diff、风险点、意图匹配度）

5. 异常兜底与方向修正（AI搞不定的问题，给方向不亲手写代码）

### Agent的角色（纯执行层，全权负责的7件事）

所有重复劳动、执行层工作全部放权，无需你动手：

1. 代码读写、实现、重构、bug修复

2. 本地校验、格式、lint、类型检查、构建

3. 测试用例补全、回归验证、CI报错修复

4. 文档同步更新、提交信息、PR说明编写

5. 方案细化、任务拆解、小问题自主闭环

6. 代码交叉审查、风险点排查

7. 命令行执行、CLI工具调用、环境适配

## 1. 不可动摇的6大核心原则

每一条都来自Peter公开的实战总结，是整套体系能跑通的根基，严禁突破。

1. **先对齐意图，再动手实现**

任何任务先讲清「目标、边界、验收标准、不做什么」，严禁一上来就让AI写代码。方案没确认，绝对不开工编码。

1. **按风险分级管控，不搞一刀切流程**

用Blast Radius（影响范围）给任务分级，小改动提速、大改动控险，不用大厂重流程束缚个人开发效率。

1. **CLI-first，轻量优先**

能用CLI解决的，绝不搞复杂MCP/中间层。Peter反复强调：模型对Unix命令的理解和执行稳定性，远胜于复杂协议封装。

1. **fix-forward，闭环优先**

接受不完美，先闭环交付，再迭代优化。小问题不反复回滚，优先往前修复，避免在细节上拉扯内耗。

1. **上下文极简干净**

不同任务、不同角色的Agent，必须分独立窗口/会话，绝不把所有内容塞进一个超长上下文，避免污染和幻觉。

1. **规则少而硬，拒绝无效制度**

只保留真正影响产能和风险的红线规则，不写一堆没人执行的空泛制度，让AI能一眼读懂、严格执行。

## 2. 双执行模式，适配全场景需求

根据任务风险、你的熟练程度，灵活选择模式，彻底解决「流程重、效率低」和「太随意、出风险」的矛盾。

||Fast Mode 极速模式|Safe Mode 安全模式|
|---|---|---|
|核心特点|小步提交、极速迭代、本地兜底|分支隔离、全流程校验、风险前置|
|适用场景|你熟悉的项目、L1小改动、低风险迭代|新项目、L3大改动、核心模块、高风险逻辑|
|分支规则|直接在main分支开发，原子化小提交|必须创建独立分支，完成后PR合并|
|校验要求|本地检查通过+你看diff放行|多Agent交叉审查+CI全绿+你终审|
|Peter实战匹配|他当前主力模式，靠小提交和diff review兼顾速度与安全|仅用于架构调整、核心链路改动等高风险场景|
新手建议：前期全程用Safe Mode练手，熟练掌握规则和Agent特性后，再逐步切换Fast Mode提效。

## 3. 核心灵魂：Blast Radius 风险分级管控

这是Peter这套体系最关键的设计，直接决定了任务的执行流程，避免「小改动走大流程，大改动控不住风险」。

### L1 小改动（低风险，极速执行）

**适用场景**：小bug修复、文案修改、单个组件微调、日志补充、样式优化、小型代码重构

**标准执行流程**：

1. 你写清极简需求+验收标准

2. 直接交给Builder Agent执行

3. Agent完成后跑本地校验

4. 你只看diff，确认无风险直接提交合并

### L2 中改动（中风险，先收敛再执行）

**适用场景**：单页面功能新增、接口接入、模块局部重构、一组测试补全、非核心功能开发

**标准执行流程**：

1. 你写清完整需求、边界、验收标准

2. 先让Agent出2-3套实现方案，你拍板最终方案

3. Builder Agent按方案编码实现，完成本地校验

4. Reviewer Agent做交叉审查，输出问题清单

5. Builder修复问题，Verifier做最终验证

6. 你终审核心diff，确认无误后放行

### L3 大改动（高风险，先计划再开工）

**适用场景**：架构调整、跨模块联动、依赖替换、数据结构重构、认证/支付/权限等核心链路改动

**标准执行流程**：

1. 你写清完整需求、影响范围、硬性约束

2. 必须先让Agent出详细实现计划，明确改动点、风险点、回滚方案、验证方法

3. 你审批计划，锁定方案和边界，强制启用Safe Mode

4. 按计划拆解子任务，分步骤执行、分阶段验证

5. 每一步都必须经过「编码-审查-验证」闭环

6. 全量改动完成后，做完整的回归验证，你终审全量变更，确认无误后合并

## 4. 最小Agent组织与全流程SOP

拒绝复杂的角色编排，只保留3个核心执行角色，对应Peter「多终端窗口并行」的实战模式，每个角色单独开一个会话/窗口，上下文完全隔离，互不干扰。

### 核心Agent角色分工

|Agent角色|核心职责|权限边界|
|---|---|---|
|Builder Agent|需求理解、编码实现、本地校验、问题修复、文档同步|仅可修改任务指定的文件，严禁越权改动架构/配置/依赖|
|Reviewer Agent|只读式代码审查，不做任何修改|只输出问题清单，不改动代码，重点查跑偏、越界、风险、漏洞|
|Verifier Agent|校验执行、测试补全、功能验证、CI修复|仅可修改测试文件、执行校验命令，不改动业务代码|
你不是第4个Agent，你是全局的**编排者+最终审批人**，只做决策和终审。

### 标准化全流程SOP（拿来就能用）

#### 前置准备（一次配置，永久复用）

1. 项目根目录放入`AGENTS.md`规则文件

2. 配置好项目的lint、typecheck、test、build等基础校验命令

3. 初始化好Git环境，配置好保护分支规则

#### Step 1：写任务卡（你唯一必须亲手写的内容）

用固定模板，把需求讲死，从根源避免AI跑偏，模板直接填空：

```Plain Text

任务名称：
Blast Radius：L1 / L2 / L3
执行模式：Fast / Safe

目标：一句话说清要解决什么问题
验收标准：列清楚什么叫「做完了」，可量化、可验证
边界与约束：技术限制、架构要求、不能碰的文件
绝对不做的事：明确排除的范围，避免过度设计
相关参考文件：需要Agent优先读取的文件路径
```

#### Step 2：方案收敛（L2/L3必做，L1可跳过）

给Agent发指令，先对齐方案，绝不直接开工：

```Plain Text

先读取项目根目录的AGENTS.md，再看相关参考文件。
不要写任何代码，先针对这个需求，给出2-3套可落地的实现方案。
每套方案必须说明：实现思路、需要改动的文件范围、潜在风险、优劣对比、推荐理由。
```

你只需要做一件事：拍板最终方案，补充调整要求，划定执行边界。

#### Step 3：Builder执行编码

给Builder Agent发确认后的方案和执行指令：

```Plain Text

按照我们最终确认的方案执行。
先严格读取AGENTS.md，遵守所有规则和红线。
只修改任务范围内的文件，严格遵循项目现有代码模式。
完成后必须：
1.  执行项目对应的format/lint/typecheck/build校验
2.  总结改动的文件清单、核心变更内容
3.  说明已执行的校验、自测结果、潜在风险点
```

#### Step 4：Reviewer交叉审查（L2/L3必做，L1可跳过）

给Reviewer Agent发指令，做前置风险拦截：

```Plain Text

你只做只读审查，绝对不修改任何代码。
先读取AGENTS.md和本次任务的需求、方案。
重点审查4个维度：
1.  实现是否偏离需求意图，有没有做超出边界的内容
2.  是否越过架构红线，有没有修改禁止改动的内容
3.  有没有安全风险、异常处理遗漏、逻辑漏洞
4.  有没有可维护性问题、不符合规范的代码
输出格式严格按以下分类：
- Blocking（必须修复的阻断性问题）
- Important（建议修复的重要问题）
- Nice to have（可选优化建议）
```

#### Step 5：问题修复与验证

把Reviewer的问题清单交给Builder Agent，指令：

```Plain Text

根据这份审查报告，修复所有Blocking和Important级别的问题。
修复完成后，重新执行所有本地校验，输出修复结果和验证情况。
```

修复完成后，交给Verifier Agent做最终校验：

```Plain Text

你只做验证工作，不做业务代码修改。
先读取任务需求和验收标准，执行对应的校验命令：
1.  执行format/lint/typecheck/build，确认全量通过
2.  验证功能是否符合验收标准，bug是否完全修复
3.  补充对应的回归测试用例，执行测试并输出结果
最终输出：验证执行情况、是否通过、剩余问题。
```

#### Step 6：你终审放行

你只需要看4件事，绝不逐行读代码：

1. 最终实现是否完全匹配需求意图，有没有跑偏

2. 核心diff有没有安全风险、架构越界、红线突破

3. 所有校验、测试是否全部通过，有没有阻断性问题

4. 有没有超出边界的改动

- 符合要求：直接合并、发布，闭环任务

- 不符合要求：明确给出修正方向，让Agent重新迭代，绝不亲手改代码

## 5. 开箱即用的核心规则与模板

### 5.1 精简版 [AGENTS.md](AGENTS.md)（直接放进项目根目录）

这是Peter风格的高压缩、真执行版本，拒绝空泛制度，只留硬规则、硬红线，AI能100%读懂并执行。

```Markdown

# AGENTS.md
## 0. 最高优先级规则
执行任何操作前，必须先完整读取本文件。
如果本文件规则与单次任务指令冲突，立即停止并向负责人确认，严禁擅自执行。

## 1. 角色定位
你是纯执行Agent，不是决策者。
严禁擅自做产品、架构、依赖、安全相关的决策，你的核心职责是实现、验证、清晰反馈。

## 2. 标准执行流程
1.  先100%理解任务意图、验收标准、边界限制
2.  先查看相关文件的现有代码，再做修改，绝不凭空写代码
3.  保持改动最小化，只改和任务相关的内容
4.  完成后必须跑通对应的本地校验，不通过绝不交付
5.  清晰报告：改了什么、为什么改、做了什么校验、有什么风险

## 3. 绝对不能碰的红线
严禁以下操作，触及任何一条立即终止执行并上报：
- 擅自修改项目架构、核心配置、基础依赖
- 未经审批新增第三方依赖包
- 修改与本次任务无关的文件/代码
- 绕过权限控制、校验逻辑、日志记录
- 硬编码密钥、Token、敏感信息、环境变量
- 执行rm -rf、git push -f、重置历史等高危命令
- 发起非业务必需的网络请求、修改项目外的文件

## 4. 编码规范
- 优先复用项目现有的模式、工具函数、组件、样式，不重复造轮子
- 遵循项目现有的命名、格式、分层规范，不发明新的写法
- 优先小改动、小提交，不做无意义的大重构
- 只在意图不清晰的地方加注释，不写废话注释
- 保持文件模块化，避免单文件代码过度膨胀

## 5. 交付硬性门槛
交付前必须跑完对应校验，不满足严禁提交：
- 代码格式化、lint检查、类型校验全部通过
- 项目构建无报错
- 修复bug必须补充对应的回归测试用例
- 功能变更必须同步更新对应的文档
- 改动说明清晰，可追溯、可验证

## 6. 必须立即上报的情况
遇到以下情况，绝不擅自决策，立即停止并说明情况：
- 需求意图、边界不清晰，可能出现理解偏差
- 改动需要突破本文件的任何规则
- 尝试2次仍无法解决的报错/问题
- 改动的影响范围超出了任务预期
- 发现现有架构/逻辑存在重大缺陷
- 涉及安全、隐私、权限相关的不确定事项

## 7. Git提交规则
### Fast Mode
- 做小而原子的提交，每个提交只对应一个逻辑改动
- 提交信息必须清晰说明改动内容，禁止无意义的update/fix
### Safe Mode
- 必须在独立分支开发，严禁直接修改保护分支
- 提交信息遵循「类型(模块): 具体改动内容」规范
- PR必须完整说明需求背景、改动内容、验证情况
```

### 5.2 全局规则复用方案（Peter同款）

实现一次编写，所有项目复用，避免重复维护规则：

1. 在本地创建全局规则目录，比如`~/agent-rules/`，把通用的`AGENTS.md`放在这里

2. 每个项目的根目录里，只放极简的`AGENTS.md`，内容如下：

    ```Markdown
    
    # AGENTS.md
    执行任何操作前，必须先读取并严格遵守全局规则：~/agent-rules/AGENTS.md
    以下是本项目的专属补充规则，优先级高于全局规则：
    1.  【项目专属架构红线、规范、命令等】
    2.  【项目核心目录结构、关键文件路径说明】
    ```

### 必选基础工具
