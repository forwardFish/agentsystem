# 通用系统工程 Agent 框架设计（无业务绑定版）

## 一、框架核心定位与设计原则

### 1.1 框架定位

本框架是**与业务解耦的通用系统工程 Agent 协同体系**，定义一套标准化的 Agent 角色、交互规范、生命周期规则，支持任意业务场景（如合规资讯源接入、业务系统开发、数据治理等）的全流程自动化落地；用户仅需定义业务规则、关键节点审批、最终结果校验，其余所有执行环节由 Agent 闭环完成，核心目标是**Agent 原生、复用性强、用户最小干预**。

### 1.2 核心设计原则（通用化约束）

|原则名称|核心约束（通用化表述）|
|---|---|
|职责单一原则|每个通用 Agent 仅负责系统工程全流程中的一个原子环节，输入/输出/依赖/阻断条件完全明确，无模糊职责|
|复用性原则|Agent 核心逻辑（流程控制、状态管理、交互规范）与业务逻辑解耦，更换业务仅需替换「业务规则集」和「产出物定义」，Agent 骨架完全复用|
|用户最小干预原则|用户仅承担 3 类角色（规则定义者、关键节点审批者、最终结果校验者），不参与任何 Agent 执行环节（任务拆分、代码编写、测试执行、部署操作等）|
|全链路可追溯原则|所有 Agent 输出必须追溯到规则版本、输入引用、执行日志、Agent 版本，无不可追溯的中间态|
|异常分类处理原则|所有失败按「瞬时异常/确定性异常/外部变化异常」精准处理，禁止无意义重试、盲修瞎改|
|可中断可恢复原则|所有关键节点支持 LangGraph Checkpoint 持久化，支持审批暂停、故障恢复、流程回退，不丢失执行状态|
|规则统一原则|全框架共享统一的状态体系、规则版本、契约标准，避免跨 Agent 规则冲突|
|工程可落地原则|所有设计可直接通过 LangGraph 实现，无概念化、不可执行的内容|
## 二、统一抽象模型（框架核心契约）

### 2.1 全局状态 Schema（通用化）

全框架所有 Agent 流转必须基于此状态结构，仅通过「引用」关联业务数据，不存储大对象，确保状态轻量化、标准化：

```JSON

{
  "task_id": "string",          // 任务唯一标识（全链路唯一，与业务无关）
  "run_id": "string",           // 单次运行实例ID（同task_id可对应多run_id）
  "graph_type": "build|runtime|governance", // 所属平面（构建/运行/治理，通用化三平面）
  "stage": "string",            // 当前执行阶段（与通用Agent职责一一对应）
  "status": "pending|running|success|fail|blocked|needs_approval", // 执行状态枚举
  "input_ref": "string",        // 输入数据引用（对象存储ID/数据库ID/文件路径）
  "output_ref": "string",       // 输出数据引用（规则同上）
  "error": "object|null",       // 错误对象：{error_type, error_msg, error_source, impact_scope}
  "retry_count": 0,             // 当前节点重试次数（控制重试策略）
  "rule_version": "string",     // 执行依据的规则版本（全链路强制携带）
  "artifact_refs": [],          // 本次执行产出物引用列表（代码包、报告、文档等）
  "approval_required": false,   // 是否触发用户审批
  "metadata": {                 // 元数据（通用化，无业务绑定）
    "start_time": "timestamp",
    "end_time": "timestamp",
    "agent_name": "string",
    "agent_version": "string",
    "trace_id": "string",
    "upstream_agent": "string"
  }
}
```

### 2.2 核心通用契约对象（与业务解耦）

框架全流程围绕以下通用对象运转，替换业务场景仅需扩展对象的业务字段，核心结构不变：

|通用对象名称|定义与用途（通用化）|所属平面|
|---|---|---|
|`UserRuleSet`|用户定义的所有规则集合（含业务规则、技术规则、测试规则、部署规则、验收阈值）|全平面|
|`ResourceRegistry`|资源注册表（通用化，可映射为任意业务资源：资讯源、接口、数据库、服务器等）|全平面|
|`RawArtifact`|原始产物对象（通用化，可映射为原始文档、原始代码、原始数据等）|Build/Runtime|
|`NormalizedArtifact`|标准化产物对象（为下游提供统一数据格式，无业务绑定）|Runtime|
|`ComplianceCheckResult`|合规/规则校验结果（含校验项、是否通过、阻断原因、风险等级）|全平面|
|`ContractSpec`|数据与接口契约规范（含各类Schema、入参出参标准，通用化）|Build/全平面|
|`BuildArtifact`|构建产出物（代码包、依赖清单、开发文档等，通用化）|Build|
|`TestReport`|测试报告集合（功能、契约、合规、可靠性测试，通用化）|Build|
|`DeployReport`|部署报告（预发/生产部署记录、环境验证结果，通用化）|Build|
|`AuditLog`|全链路审计日志（操作人、操作内容、时间、结果，通用化）|全平面|
|`FinalGateDecision`|最终放行决策报告（通用化决策结论，无业务绑定）|全平面|
|`ChangeRequest`|用户变更请求（通用化，含变更内容、原因、新规则、生效时间）|全平面|

所有通用 Agent 必须遵循此输出结构，核心内容结构化，禁止自然语言长文，替换业务仅需修改 `content` 字段的业务逻辑：

```JSON
  "rule_version": "string",
  "status": "success|fail|blocked|needs_approval",
  "content": "object", // 结构化核心输出（业务逻辑层，框架不约束）
  "error": "object|null",
  "artifact_refs": [],
  "metadata": {
    "start_time": "timestamp",
    "end_time": "timestamp",
    "trace_id": "string",

  }

```

## 三、通用 Agent 组织体系（核心框架）

框架将系统工程全流程拆解为**无业务绑定的通用 Agent 角色**，每个 Agent 定义明确的职责、边界、输入输出、阻断规则，替换业务场景仅需修改「业务规则集」和「产出物校验逻辑」。

### 3.1 总体组织架构（通用化）

```Plain Text

通用系统工程Agent框架
├── Build Plane Agents（构建平面：功能从0到1落地）
├── Runtime Plane Agents（运行平面：已交付功能7×24运行）
├── Governance Plane Agents（治理平面：全流程规则约束与审计）
├── Final Gate Agent（最终放行决策：通用化结果审核）
└── Human Approval Checkpoint（人工审批关口：唯一用户干预点）
```

### 3.5 Final Gate Agent（通用最终决策）

#### 核心定位

框架唯一的最终放行决策中枢，通用化决策逻辑，无业务绑定，是用户校验结果的唯一入口。

#### 核心职责（通用化）

1. 汇总 Build/Runtime/Governance 三平面全量结果；

2. 对照 UserRuleSet 逐条核验执行情况；
