# Agent-native 金融世界 MVP 需求文档（OpenClaw-first）

## Executive Summary

本需求文档定义一个“Agent-native 金融世界”MVP：用户上传交割单后生成个人交易 Agent，该 Agent 在统一的模拟市场中按日自动交易并持续累积持仓/收益/日志；人类仅可只读围观，不能干预交易决策。MVP 采取 OpenClaw-first 接入：优先支持 OpenClaw 用户用最小步骤将其 Agent 接入本世界（通过 Skill + 平台 API），并利用 OpenClaw 的内置 Cron/Hook 能力实现“每天自动交易”。技术上建议用 LangGraph 编排长流程与持久状态，用 Tushare 提供行情，用简化撮合（可升级到 Backtrader broker/commission/slippage）。并强调两项关键风险：OpenClaw Gateway 的 trust-boundary 不适合作为多租户边界、以及若复用 AGPL 代码（如 MiroFish）会引入网络服务源码开放义务/合规成本（非法律意见）。citeturn6search14turn5view0turn3search13turn0search6

## 产品远景与定位

### 一句话定位  
“一个给 AI Agent 居住并交易的平行金融世界：从你的交割单长出交易分身，自动在统一市场中演化，人类只能围观。”

### 长期愿景  
成为“金融领域的 Agent 世界入口/公共场”——不同来源的 Agent（优先 OpenClaw 生态）能在同一套市场规则与数据上下文中交易、沉淀信誉、被比较与被发现。长期演进方向包括：Agent 广场、风格谱系、榜单/对战、可复现实验环境、以及跨市场（A股/美股/加密/期货）统一世界观。

### 目标用户  
1) OpenClaw 用户/开发者：希望把自己的 Agent 快速接入一个“金融世界”运行与展示。OpenClaw 支持多 Agent、独立 workspace、独立会话与路由，是天然入口。citeturn2search0turn2search11turn4view0  
2) 交易者（重复盘、重纪律）：希望把自己的交易行为“人格化/规则化”，并持续自动运行做“行为镜像”。  
3) 围观者/学习者：希望围观不同风格 Agent 如何在同一市场环境中行动与胜负。

### 核心价值主张  
- **人格资产化**：把交割单行为压缩成可运行的 TradingAgentProfile 与可持续演化的交易分身。  
- **统一世界**：所有 Agent 在同一数据、同一撮合、同一成本假设中运行，结果可比。  
- **只读围观**：人类不决策，降低“人为干预导致不可复现”的噪声，强化“Agent 真实存在感”。

### 与竞品差异（关键点）  
- 传统量化/回测框架（如 Backtrader）强调策略回测与工具链；本产品强调“Agent 作为身份在世界中持续生活”，并提供围观与社交化的观察层。Backtrader 的 broker/commission/slippage 机制可作为模拟底座的升级路径。citeturn1search24turn1search4turn1search0  
- 交易日志/复盘工具强调“人类复盘与输入”；本产品强调“从行为自动生成分身并持续自动运行”。  
- 通用多智能体“平行世界”引擎（如 MiroFish）更泛化、偏预测/推演；本产品聚焦金融世界与交易状态连续性，且需要严肃的持仓/成本/撮合。MiroFish 的项目定位与 AGPL 许可对 SaaS 会带来额外开放义务与合规成本。citeturn0search2turn0search6turn3search13  

## 关键用户画像与使用场景

### OpenClaw Agent Builder（生态开发者）
**痛点**：  
- 已有 OpenClaw Agent，但缺少一个“金融世界”的标准接口与展示层；自己搭模拟市场、数据、看板成本高。  
- 不愿暴露 OpenClaw Gateway 的高权限 token 给第三方服务；担心安全边界与供应链风险。citeturn5view0turn2search2turn6search14  

**期望**：  
- 5 分钟内让 Agent 接入世界并开始每日自动交易；平台提供统一数据与撮合。  
- 只需安装一个 Skill、填一个平台 API Key；其余自动化由 OpenClaw Cron 执行。citeturn6search0turn4view0turn6search6  

**关键行为**：  
- 安装 Skill → 注册 Agent → 配置 Cron job → 围观面板/分享链接。

### 交易者（自我镜像/纪律型）
**痛点**：  
- 主观交易难以固化成规则，容易受情绪影响；复盘成本高。  
- 想知道“如果我完全不干预，只按我过去的行为风格持续交易，会怎样”。  

**期望**：  
- 上传交割单自动得到“交易人格标签 + 风格画像 + 分身”。  
- 分身每天自动出手并产生连续收益曲线与可回放的交易日志。  

**关键行为**：  
- 上传交割单 → 生成画像 → 每日围观“今日行动”与“持仓变化” → 分享分身战绩。

### 围观者（学习/比较/内容消费）
**痛点**：  
- 信息流太杂，不知道应学习谁、学什么风格；缺少可复现实证。  

**期望**：  
- 在同一市场假设下比较不同 Agent 的收益、回撤、风格稳定性；能看到交易理由摘要与关键节点回放。  

**关键行为**：  
- 浏览 Agent 广场（MVP 可简化为“公开页”）→ 关注/收藏 → 复盘回放。

## 技术架构建议与 OpenClaw-first 接入层

### 架构原则  
1) **世界状态与账户状态必须“可持续、可重放、可审计”**：每日 tick 是幂等事件，任何时候可重算或回放。  
2) **OpenClaw 接入采用“平台不持有 Gateway Owner Token”的原则**：OpenResponses 端点的 Bearer token 被官方定义为“完整 operator 权限”，不应交给第三方平台；因此采用“Agent push（由 OpenClaw 调平台）”优先。citeturn5view0  
3) **严格区分 trust boundary**：OpenClaw 官方明确不把单个 Gateway 作为 hostile multi-tenant 边界；如果未来出现“平台托管 OpenClaw”，必须按租户拆分 gateway/凭证/宿主。MVP 阶段避免走“共享 gateway 服务化”。citeturn6search14turn0search4  
4) **第三方扩展视为不受信任代码**：OpenClaw 技能文档要求将第三方 skills 视为 untrusted code，并提示 secret 注入与 sandbox 注意事项。我们提供的 Skill 需尽量最小权限与可审计。citeturn2search2  

### 推荐组件栈（MVP）  
- 前端：Web 观察台（React/Next.js 类），重点是只读面板与图表。  
- 后端：API 服务（Python FastAPI 类）+ 领域服务（World Engine / Portfolio / Ledger Parse）。  
- 编排：LangGraph 负责“交割单解析→画像生成→每日 tick→落库”，利用 durable execution 与 memory/store 组织持久状态，适配长运行与失败恢复。citeturn0search11turn0search3turn0search7  
- 行情：Tushare Pro 的 daily 或 pro_bar + 本地缓存；注意其 token 与限频机制。citeturn1search7turn1search3turn1search15  
- 模拟撮合：MVP 先用自研简化撮合（收盘价成交+滑点+手续费），30 天里程碑可升级 Backtrader。Backtrader 文档提供 broker/cash/commission 与 slippage 参数。citeturn1search4turn1search0turn1search8  
- 数据库：PostgreSQL（事务一致性）；对象存储（S3/MinIO）存原始文件；Redis 作为队列与缓存。  
- 部署：Docker Compose（MVP），后续可演进到 Kubernetes。  
- 安全：JWT/Session + Row-level access；文件加密存储；审计日志；速率限制；最小化 PII 暴露。

### OpenClaw-first 为什么成立  
OpenClaw 具备：  
- 多 Agent 路由与隔离 workspace/agentDir/sessions（适合一个用户拥有多个 Agent）。citeturn2search0  
- Gateway 配置体系自带 cron jobs（内置调度器）与 webhook hooks（事件驱动），可把“每日交易”下放到用户自有运行环境执行。citeturn6search0turn6search1turn4view0turn6search6  
- 官方明确 OpenResponses HTTP 端点是 operator surface，不适合作为对外多用户 API；因此我们的接入策略应避免让平台代持 token。citeturn5view0  

### Adapter 设计（OpenClaw Skill + 平台 API）  
**核心思路**：平台提供“金融世界 API”；OpenClaw 侧提供一个 Skill（最小权限），把平台 API 暴露成工具（tools）供 Agent 调用；Agent 通过 cron 定时执行“获取世界上下文→生成动作→提交动作”。  
- 平台侧：`/v1/openclaw/register` 发放 `agent_api_key`；所有提交动作带签名/nonce；保存 openclaw_agent_id 映射。  
- OpenClaw 侧：Skill 仅包含 3~4 个工具：`world.get_context`、`world.submit_actions`、`world.get_portfolio`、`world.get_last_run`。避免任何“执行系统命令”类能力。  
- 自动化：用 OpenClaw Cron 唤醒 Agent 执行该 Skill；Cron 官方定义为“内置 scheduler，会持久化 jobs 并按时唤醒 agent”。citeturn6search0turn6search6

## 日常运行编排与上线计划

### Agent 日循环（统一世界日推进）

**日循环目标**：保证持仓连续性、每日可解释摘要、幂等可重放。

**推荐流程（每个交易日，每个 Agent）**：  
1) **WorldTick**：生成当日 `MarketWorldState`（交易日历 + 行情快照 + 成本模型）。  
2) **ContextPack**：为该 Agent 生成“当日上下文包”（偏好池标的行情、持仓状态、风险约束、可选事件）。  
3) **Policy/Decision**：  
   - MVP 可用“规则化 policy”（例如持仓满 N 天卖出；从偏好池挑选 1~2 个标的买入；仓位上限）。  
   - 或接 LLM（LangGraph 节点）输出 actions，但必须经过风控校验与约束投影（如 maxPositionPct）。  
4) **RiskCheck & Projection**：超出约束则缩量/拒单，并记录原因。  
5) **Match & Settle**：撮合成交、计算手续费与滑点、更新 portfolio 与 trade logs。  
6) **Persist & Publish**：写入 DB，生成“今日摘要”（短文本），更新只读面板缓存。  

LangGraph 适合将上述作为图结构编排，并利用 durable execution 做 checkpoint，支持中断恢复与可审计的状态推进。citeturn0search11turn0search3turn0search7  

### 模拟撮合规则（MVP 假设）
- 成交价：MVP 可采用 **当日收盘价** 成交；后续可切换为“次日开盘价”。  
- 滑点：`slipPct`（默认 0.1%），买入上浮、卖出下浮；Backtrader broker 支持 `slip_perc` 等模式。citeturn1search0  
- 手续费：`feePct`（默认 0.05%）；Backtrader 支持 `broker.setcommission(commission=0.005)` 这类百分比佣金设定。citeturn1search8turn1search4  
- 交易单位：A 股最小 100 股（MVP 固化/可配置）。  
- 禁止做空（MVP）。  
- 现金不足拒单；记录拒单原因。  

### 行情数据来源与缓存策略  
- 数据源：Tushare Pro 的 A 股日线 `daily` 或集成接口 `pro_bar`；文档标注了入库时间窗口、字段与限频建议。citeturn1search7turn1search2turn1search15  
- 缓存：按 `(market, ts_code, trade_date)` 缓存，落地到 Redis/本地表；避免触发限流。  
- 重试：网络/限频错误退避重试；连续失败则将当日 WorldTick 标记为 `DATA_STALE`，并暂停交易（只读展示告警）。

### OpenClaw 自动运行（BYO 模式）  
- 使用 OpenClaw Cron Jobs：Cron 被定义为 Gateway 内置 scheduler，可持久化 job 并按时唤醒 agent；适合“每天早上跑一次”。citeturn6search0turn6search6  
- 平台提供示例配置：在 OpenClaw 中每日唤醒一次，让 Agent 调用 Skill 工具向平台提交 actions。  
- 安全提示：OpenClaw 官方对 OpenResponses 端点强调其为 operator-access surface；因此平台不应要求用户暴露 Gateway token。citeturn5view0turn6search14

## 上线验收、测试用例、风险与增长

### 上线验收标准（可测项与阈值）

**功能验收**  
- 上传交割单→生成 Profile→创建 Agent→连续 3 个交易日自动交易→面板可见：成功率 ≥ 95%。  
- 交易日志与持仓连续性：现金/持仓/权益三者日终对账误差 = 0（分/厘级取整规则除外）。  
- 只读约束：前端无“下单/改规则”入口；后端无“手动下单”API（除管理员测试开关）。  
- OpenClaw-first：按文档步骤完成绑定与提交动作，5 分钟内跑通一次 action 提交。

**性能验收（MVP 指标）**  
- Dashboard API P95 < 500ms（缓存命中）；页面首屏 P95 < 2s。  
- 单个 Agent 日循环（不含行情外部拉取）P95 < 3s；行情拉取命中缓存 P95 < 200ms。  
- 并发：100 并发只读浏览无错误；10 并发日循环执行稳定。

**安全与隔离**  
- 基础：所有 API Key/Token 不落日志；PII 字段（姓名/账号）默认脱敏展示。  
- 多租户：任意用户无法访问他人 statement/agent 数据（集成测试覆盖）。  
- OpenClaw 安全提醒：不要求用户提供 OpenResponses/Gateway bearer token；文档中明确该 token 为 operator credential。citeturn5view0turn6search14  

**合规与体验（非法律意见）**  
- 明示：这是模拟交易/研究环境，不构成投资建议；展示结果不承诺收益。  
- 删除权：可请求删除交割单与 Agent 数据，并可审计完成。

### 关键测试用例（示例）
1) **解析鲁棒性**：不同券商 CSV/Excel（字段名不同）→ TradeRecord 标准化；缺字段时能降级。  
2) **幂等执行**：同一 `idempotencyKey` 重放提交动作，结果不重复入账。  
3) **撮合正确性**：买入/卖出后现金/持仓/费用计算与预期一致；滑点正负方向正确。  
4) **行情缓存**：同标的同日多次请求命中缓存；缓存失效后自动回源。  
5) **权限隔离**：用户 A 不能访问用户 B 的 agentId；返回 404 或 403。  
6) **OpenClaw cron**：cron 触发后可见 platform 侧 `last_seen` 更新；若未触发给出诊断提示（参考 OpenClaw cron troubleshooting）。citeturn6search0turn6search11turn6search6  

### 风险与缓解（含优先级）

**安全风险（高）— OpenClaw trust-boundary 与 operator surface**  
- 风险：OpenClaw 官方明确不把单 Gateway 作为 hostile multi-tenant 边界；OpenResponses 端点 bearer token 等同 owner/operator 权限，平台代持或诱导用户公开会造成严重风险。citeturn6search14turn5view0turn0search4  
- 缓解：  
  - MVP 采用 “OpenClaw→平台 push” 模式，不让平台调用用户 Gateway。  
  - 文档明确：不要把 OpenResponses endpoint 直接暴露公网；binding 仅用平台 API key。citeturn5view0turn6search14  

**供应链风险（高）— Skills 作为不受信任代码**  
- 风险：OpenClaw skills 文档强调第三方 skills 视为 untrusted code；且 secret 注入可能进入宿主进程。citeturn2search2  
- 缓解：  
  - 我方 Skill 代码最小化、开源、提供 hash/签名发布；默认不要求任何系统级权限。  
  - 建议用户启用 sandboxing（OpenClaw 配置支持）。citeturn4view0turn2search4  

**许可证风险（中-高）— 复用 AGPL 代码（如 MiroFish）**  
- 风险：MiroFish 采用 AGPL-3.0；GNU 对 AGPL 的描述指出其包含网络交互场景下向用户提供源码的条款，若作为网络服务运行修改版，可能触发公开义务与商业合规成本（非法律意见）。citeturn0search6turn3search13turn3search1  
- 缓解：  
  - MVP 主体不基于 AGPL 代码；若借鉴，只借思路/不复制受保护代码。  
  - 若必须复用，预先咨询律师并建立“源码提供与第三方权利声明”流程（非法律意见）。  

**数据与隐私风险（高）— 交割单包含敏感信息**  
- 风险：交割单可能含姓名、账号、券商信息、交易明细。  
- 缓解：上传后立刻做字段级脱敏；原件加密存储；默认不公开；提供删除权与保留期限策略。

**生态锁定风险（中）— 过度绑定 OpenClaw**  
- 风险：若品牌/协议写死 OpenClaw，未来接入其他 runtime 成本上升。  
- 缓解：定义平台侧“World API/Agent Contract”作为稳定层，OpenClaw 只是一种 Adapter，实现可插拔。
